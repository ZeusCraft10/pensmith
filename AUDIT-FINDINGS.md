# Pensmith — QA Audit Findings

_Generated 2026-06-26. Method: the 971-test unit suite (all green) + a clean-room offline end-to-end CLI run + an 11-subsystem multi-agent bug hunt (153 agents) in which every candidate was adversarially verified by 3 independent skeptics; only findings surviving at least 2 of 3 votes are listed. Scoreboard: 3 CRITICAL / 11 HIGH / 18 MEDIUM / 5 LOW._

## Manual findings (hand-verified)
- **M1 — research/router sentinel mismatch.** `research` writes `LIBRARY.json`, but the router (`bin/lib/router.ts:169,175`) and the list/status deriver (`bin/lib/global-library.ts:358`) gate "research done" on `RESEARCH.md` (a different, later artifact). Bare `pensmith`/next/resume loop on research.
- **M2 — write/compile raw stack traces.** A section-less `OUTLINE.md` makes `write`/`compile` dump unhandled `parseOutline` stack traces; `done` degrades gracefully by contrast.
- **M3 — global registry never GC'd.** `%LOCALAPPDATA%/pensmith/library/index.json`: 344 entries, 158 (46 percent) dead; tests pollute the real registry (no LOCALAPPDATA isolation).

## Two systemic themes
**A. The single-command pipeline cannot advance (state-transition disconnect).** The Tier-2 CLI verbs do not write the STATE.json / PLAN.md transitions the router reads, so bare `pensmith` / `next` / `resume` loop at each stage: outline never registers `state.sections` (#1), write never sets `status: written` (#9), verify never sets verified/failed (#8), done is never terminal (#15) — plus M1. The headline single-command UX is non-functional end-to-end in Tier 2. The unit tests pass because each verb is tested with pre-seeded state.

**B. The verifier gate (the number-one safety non-negotiable) has confirmed bypasses.** FABRICATED/MIS-CITED enforcement lives only in `compile`; `done`/export does not re-run Pass-1 and skips GATE-04 under `--raw`/offline (#3, #14). The Pass-1 citekey regex misses uppercase/locator/multi-cite keys, so those citations are invisible to the verifier and pass vacuously (#2, #20). Stored-retracted (`note=RETRACTED`) flags are never read by Pass-1 (#6). Net: fabricated/mis-cited/retracted citations can reach the exported manuscript.

### 2. Uppercase / locator / multi-cite citekeys are invisible to the verifier — FABRICATED & MIS-CITED citations escape compile and export
- **Severity:** CRITICAL · **Adversarial votes:** 3/3 · **Subsystem:** verifier-gate
- **Location:** `bin/lib/citation-token.ts:38 (shared regex); bin/lib/verify/pass1.ts:211; bin/lib/verify/verdict-rows.ts:77; bin/cli/done.ts:363`
- **What / why:** The verifier's contract (PROJECT.md non-negotiable #4) is that no FABRICATED/MIS-CITED citation may escape the gate. Here a citation is treated as 'absent / nothing to verify' instead of 'unparseable / fail-closed'. The narrower extraction regex is silently a safety hole: a fabricated reference that the regex cannot parse produces NO verdict at all, so it passes every gate. Pandoc locator ([@k, p. 5]) and multi-cite ([@a; @b]) are completely standard academic citation forms (citation-token.ts:10-12 itself admits they are 'OUT OF SCOPE'), and uppercase keys are the dominant BibTeX convention (Smith2017-style), reachable via hand-edited drafts, imported .bib files, or a drafter slip. Because extractCitekeys (compile bib regen) and done.ts GATE-04 use the same regex, the token is also dropped from the regenerated CITATIONS.bib and from the final-MD citekey-diff check, so the exported manuscript ends up with a dangling, unverifiable citation and no bib entry. The 'done' export path never re-runs Pass-1 at all, so compile's (bypassed) gate is the only Pass-1 enforcement.
- **Repro:** RUNTIME-CONFIRMED offline. Temp .paper with CITATIONS.bib containing only lowercase smith2020; section DRAFT.md = 'Prior work established the baseline [@Vaswani2017], which we build upon here.' (Vaswani2017 NOT in bib). `pensmith verify 1 --slug intro` -> writes 'Status: verified' with an EMPTY Pass-1 section (no FABRICATED row). `pensmith compile --yolo` -> succeeds (refuse_reasons: []), writes .paper/DRAFT.md still containing [@Vaswani2017], and regenerates an EMPTY CITATIONS.bib. Separately, a draft with only [@madeupghost2099, pp. 10-15] (fabricated, locator form) also verifies clean with zero Pass-1 rows. Compare: the same fabricated key in bare-lowercase form ([@madeupkey2099]) IS caught as FABRICATED — proving the regex shape is the sole cause.

### 7. Pass-2 / Pass-4 LLM verifier instantiate the Anthropic SDK directly, bypassing the http.ts (D-06) transport chokepoint
- **Severity:** HIGH · **Adversarial votes:** 3/3 · **Subsystem:** llm-budget-http-security
- **Location:** `bin/lib/verify/pass2.ts:29 (value import) & :251 (new Anthropic) & :281 (client.messages.create); bin/lib/verify/pass4.ts:57 & :444 & :469`
- **What / why:** Every other LLM call site routes through complete() → http.ts, which applies the SSRF pre-flight guard, full-jitter retry/backoff, the polite User-Agent, and central transport handling. pass2/pass4 get NONE of these: no SSRF guard, no retry on 429/5xx, no User-Agent contact header, and the request never passes the chokepoint the security model assumes is total. The ESLint chokepoint only bans `undici`/`http`/`https` imports (eslint.config.js) — it does not ban `@anthropic-ai/sdk` as a value import, so this violation is invisible to lint. It is also invisible to the test suite because pass2/pass4 short-circuit on the noLlm guard (PENSMITH_NO_LLM=1 or no ANTHROPIC_API_KEY) and CI never sets a real key. The bypass only activates for a real user with a key set, where the verifier is doing the security-sensitive work.
- **Repro:** Static: grep shows `new Anthropic(` only in pass2.ts:251 and pass4.ts:444; anthropic.ts:24 uses `import type`. The two pass files use a bare value import. eslint.config.js restricted-imports lists undici/http/https but not @anthropic-ai/sdk, so `npm run lint` does not flag the SDK instantiation.

### 10. Section-scoped verbs `plan`/`verify` invoked without N silently run a DIFFERENT verb
- **Severity:** HIGH · **Adversarial votes:** 3/3 · **Subsystem:** cli-flags-dispatch
- **Location:** `bin/pensmith.ts:313-352`
- **What / why:** `isSectionVerbWithoutNumber(argv,'plan'|'verify')` returns true for a numberless invocation, so dispatch() abandons the explicit-verb path and falls into the bare-router branch (lines 329-352). It then calls `resolveNextAction(...)` (router.ts:142) which returns the FIRST non-verified section's verb regardless of what the user requested, and `dispatchVerb(decision.verb, ...)` runs THAT verb. So `pensmith verify` on a section whose PLAN.md status is `writing` runs `write` (generating an LLM-backed DRAFT.md); `pensmith plan` on a `written` section runs `verify` and writes VERIFICATION.md; `pensmith plan` on an all-verified paper dispatches `compile`. The user asked for one action and a different, side-effecting action executes. Runtime-confirmed offline: (a) `pensmith verify` with §1 status=writing printed `pensmith write: wrote DRAFT.md ...` and created DRAFT.md; (b) `pensmith plan` with §1 status=written printed `pensmith verify: ... wrote ... VERIFICATION.md`; (c) `pensmith plan` on an all-verified paper dispatched `compile`. The comment at lines 322-326 even asserts this path 'dispatch[es] via the shared helper' but never re-checks that the resolved verb equals the requested verb. The correct behavior for a numberless section-scoped verb is to resolve the next section that NEEDS that specific verb (or error), not to hand control to the global next-action resolver.
- **Repro:** Offline, in an isolated workdir: create STATE.json with sections=[{n:1,slug:'intro'}], touch .paper/RESEARCH.md and .paper/OUTLINE.md, write .paper/sections/01-intro/PLAN.md with frontmatter `status: writing`. Run `PENSMITH_NO_LLM=1 PENSMITH_NETWORK_TESTS='' tsx bin/pensmith.ts verify`. Observed: it ran `write` and created DRAFT.md instead of verifying. Likewise `status: written` + `pensmith plan` runs verify; all-verified + `pensmith plan` dispatches compile. Confirmed at runtime.

### 11. `--dry-run` does not zero all egress: `pensmith add <url>` makes a live network request
- **Severity:** HIGH · **Adversarial votes:** 3/3 · **Subsystem:** cli-flags-dispatch
- **Location:** `bin/cli/add.ts:205`
- **What / why:** Offline gating in this codebase is implemented per-adapter: each source adapter (crossref/openalex/arxiv/pubmed/s2/unpaywall/retraction-watch) and plagiarism.ts (plagiarism.ts:250) and honesty.ts (honesty.ts:287) check `isOfflineMode()` BEFORE calling httpFetch. `bin/lib/http.ts::fetch` itself never consults `isOfflineMode()` (grep of http.ts for isOfflineMode/PENSMITH_NETWORK_TESTS finds zero gating). The `add` URL path (add.ts:204-218) calls `httpFetch(source,{source:'generic'})` directly with NO offline guard, so under `--dry-run` it proceeds to the cache→SSRF-DNS-preflight→undici `request()` dispatch (http.ts:649-686). The SSRF preflight performs a real DNS resolution and undici opens a real socket — both are external egress that `--dry-run` promises to suppress. Runtime-confirmed offline: `pensmith add https://pensmith-dryrun-probe.invalid/paper.html --dry-run --yolo` entered the network path and failed with 'could not hydrate' only because the .invalid TLD never resolves (the DNS query itself left the machine); a real host would have completed an actual HTTP fetch despite `--dry-run`.
- **Repro:** Offline: `PENSMITH_NO_LLM=1 PENSMITH_NETWORK_TESTS='' tsx bin/pensmith.ts add 'https://pensmith-dryrun-probe.invalid/paper.html' --dry-run --yolo`. The add URL branch is entered and a DNS/socket attempt is made (output: 'pensmith add: could not hydrate ... Source NOT added'). With a resolvable host this is a live HTTP request under --dry-run. Confirmed at runtime; http.ts has no isOfflineMode gate.

### 14. `done` does not require sections to be verified or DRAFT.md to be a fresh compile output — exports on any DRAFT.md
- **Severity:** HIGH · **Adversarial votes:** 3/3 · **Subsystem:** prd-noneg-drift
- **Location:** `bin/cli/done.ts:487-501`
- **What / why:** PRD §7.9/§7.8 make compile the gate that 'Refuses if any section has FABRICATED, MIS-CITED, or quote-NOT_FOUND' and write DRAFT.md only when clean; done is supposed to re-assert this ('Refuses to run if any section's verification is unclean'). Combined with findings #1/#2, the export path has no defense at all.
