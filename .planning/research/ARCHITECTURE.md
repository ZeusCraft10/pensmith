# Architecture Research

**Domain:** Claude Code plugin + portable Node CLI sharing markdown workflow bodies as source of truth, applied to academic-paper authoring with section-isolated state and bounded per-section verification.
**Researched:** 2026-05-06
**Confidence:** HIGH (gsd-plugin source inspected directly at `/tmp/refs/gsd-plugin/`; PRD §4/§13/§14 are explicit; section-as-phase load-bearing model already locked).

---

## 0. Executive summary

Pensmith is a near-direct adaptation of `jnuyens/gsd-plugin`'s architecture, with three substantive twists:

1. **Section-as-phase**, not feature-as-phase. The roadmap analogue (`OUTLINE.md`) is generated by an LLM step, not the user. Each section gets its own directory exactly like a GSD phase gets its own `.planning/phases/<N>/`.
2. **Two-tier from one source of truth.** GSD-plugin already does this through `<runtime_note>` and `--text` mode — pensmith generalises that pattern via `<capability_check>` blocks that probe `Task` / `MCP` / `AskUserQuestion` / `Pandoc` / `Zotero MCP` / `humanizer` and fork inside the same workflow body.
3. **Verifier as gating contract.** GSD's `verify-work` is advisory; pensmith's verifier is a hard gate that refuses compile/export. This forces deterministic identifier/quote checks (Node code, not LLM) to live in `bin/lib/verifier.js` next to schema-versioned state, not in agent prompts.

The system is structured as **three concentric rings**:

- **Ring 1 — Foundation NFRs** (paths, http, lock, doi, budget, schema/migrations, atomic write, session log). Everything else depends on these.
- **Ring 2 — Domain libraries** (sources, pdf-ingest, verifier, plagiarism, style-match, citations, disciplines, ecosystem, honesty, runtime).
- **Ring 3 — Workflow surface** (skills + workflow bodies + agents + MCP server + hooks + templates).

Build order is strictly outside-in on dependency, inside-out on user value: Ring 1 first, then a vertical slice through Ring 2+3 for *one* section (intake → research → outline → plan-section → write-section → verify-section), then breadth.

### Component responsibilities

| Component | Responsibility | Tier | Implementation |
|-----------|---------------|------|----------------|
| Skills (`skills/<name>/SKILL.md`) | Frontmatter (name, description, allowed-tools, argument-hint) + `@${CLAUDE_PLUGIN_ROOT}/workflows/<name>.md` include | T1 | Markdown with YAML frontmatter |
| Workflow bodies (`workflows/*.md`) | Step-by-step instructions; `<capability_check>` forks; the *only* prose source of truth | T1+T2 | Markdown with structured XML-ish blocks |
| Agents (`agents/*.md`) | Subagent prompt + allowed-tools (Task-spawnable in T1; in T2 a "subagent" is a sequential LLM call with the same prompt) | T1+T2 | Markdown with frontmatter |
| Templates (`templates/*.md`) | Skeleton output files (PLAN.md, RESEARCH.md, etc.) — referenced by both tiers | T1+T2 | Markdown with placeholders |
| MCP server (`mcp/server.js`) | Read-only state resources + idempotent state-mutation tools; **NO business logic** | T1 only | Node stdio JSON-RPC, thin wrapper over `bin/lib/*` |
| Hooks (`hooks/hooks.json`) | SessionStart resume, PreCompact handoff, PostToolUse checkpoint, Stop cleanup | T1 only | Node one-liners shelling into `bin/pensmith-tools.js` |
| Tier-2 CLI (`bin/pensmith-cli.js`) | Verb dispatcher; reads workflow bodies, executes sequentially against OpenAI-compatible endpoint; stdin prompts replace `AskUserQuestion` | T2 only | Node script |
| `bin/lib/state.js` | `.paper/STATE.md` atomic read/write; section-state aggregation | T1+T2 | atomicWriteFileSync (write-tmp-then-rename) |
| `bin/lib/checkpoint.js` | HANDOFF.json writer; **never throws** (5s PreCompact budget); per-section progress | T1+T2 | mtime-throttled |
| `bin/lib/lock.js` | PID + start-time lock file; stale auto-clear after longest-step timeout | T1+T2 | flock-style sentinel file |
| `bin/lib/paths.js` | XDG / %APPDATA% / ~/Library resolution for library + cache | T1+T2 | platform-conditional |
| `bin/lib/http.js` | Cached client, exponential backoff w/ jitter, polite UA, per-source TTL | T1+T2 | wraps `fetch` |
| `bin/lib/doi.js` | Identifier normalisation (DOI/arXiv/PMID) — single chokepoint | T1+T2 | pure functions |
| `bin/lib/verifier.js` | DOI integrity + author/title fuzzy + claim-support orchestration + quote-verify + per-¶ audit | T1+T2 | mix: deterministic (Pass 1, 3) + LLM-judged (Pass 2, 4) |
| `bin/lib/budget.js` | Per-step cost estimate, running meter, hard cap abort | T1+T2 | wraps every LLM call |
| `bin/lib/migrations/` | One file per `<from>-to-<to>` schema bump; empty in v0.1.0 but the directory ships | T1+T2 | autoload + sequential apply |

### Structure rationale

- **`bin/lib/` is the only place business logic lives.** Both tiers, the MCP server, and the hooks call into the same library. No logic in `mcp/server.js`. No logic in `skills/*/SKILL.md`. (Mirrors gsd-plugin: `mcp/server.cjs` is a thin shim over `bin/lib/*.cjs`.)
- **`workflows/*.md` is the only place prose-instructions live.** When a skill's behaviour changes, it changes in exactly one file — both tiers pick it up. (Mirrors gsd-plugin's `@${CLAUDE_PLUGIN_ROOT}/workflows/<name>.md` include pattern from `skills/*/SKILL.md`.)
- **`agents/*.md` carries subagent prompts even though Tier 2 has no `Task` tool.** In Tier 2 the dispatcher reads the same agent file and runs it as a sequential LLM call with the same context envelope. This keeps tier drift impossible by construction.
- **`schema/` is versioned, not generated.** Every state file declares `schema_version`; migrations are just JS files keyed by `<from>-<to>`. Day-1 migration directory is empty, but the loader exists.
- **`references/section-as-phase.md` is required reading.** The PRD calls it load-bearing; encoding it in `references/` and `@`-including it at the top of every section workflow makes future contributors hit it before they touch anything.

## 3. Architectural patterns

### Pattern 1: Workflow body as shared source of truth (`<capability_check>` forks)

**What:** Each workflow body is a single markdown file with explicit blocks declaring required reading, capability checks, process steps, and output. Both Tier 1 (Claude Code with Task/MCP/AskUserQuestion) and Tier 2 (Node CLI sequential) read the same file and fork on capabilities.

**Why:** GSD-plugin already does the hard part of this — text-mode for runtimes without `AskUserQuestion`. Pensmith generalises it to all rich tools.

**When to use:** Every command. No exceptions.

**Trade-offs:**
- (+) Zero duplication. Drift between tiers is structurally impossible.
- (+) Tier-2 contributors don't need to think about Tier 1 and vice-versa for body changes.
- (−) Workflow files get long; need progressive-disclosure (see GSD-plugin's `discuss-phase/modes/*.md` overlay pattern).
- (−) Capability-check authoring is delicate; needs `references/runtime-contract.md` to stay rigorous.

**Capability-check block convention** (recommended; consistent with gsd-plugin's `<runtime_note>`):

```markdown
<capability_check>
**Required:** none unconditionally — see fork below.
**Probes:** Task, MCP(pensmith), AskUserQuestion, Pandoc, Zotero MCP, humanizer skill.
**Cache:** `.paper/CAPABILITIES.json` for the run; force re-probe with `--reprobe`.

| Capability | Tier-1 path | Tier-2 fallback |
|---|---|---|
| `Task` | spawn N subagents in parallel | sequential loop with same agent prompt |
| `MCP(pensmith)` | use `paper_*` tools for state mutations | call `bin/lib/state.js` directly |
| `AskUserQuestion` | structured options dialog | numbered stdin prompt (matches gsd-plugin `--text` overlay) |
| `Pandoc` | export `.docx`/`.pdf` via pandoc | export `.md` only; warn cleanly |
| `Zotero MCP` (auth-checked) | offer as additional source provider | skip provider; note in RESEARCH.md |
| `humanizer` skill | invoke `~/.claude/skills/humanizer/` | print "humanizer skill not present, skipping" and continue |
</capability_check>
```

Workflow steps that branch on capabilities use:

```markdown
**If `Task` available:**
   Spawn one `pensmith-source-researcher` per query in parallel via Task tool.
**Else (Tier 2 / Task unavailable):**
   Loop sequentially through queries, one runtime.js call each.
   The agent prompt is identical (`agents/pensmith-source-researcher.md`).
```

This keeps the *prompt* and the *output contract* identical across tiers — only orchestration differs.

### Pattern 2: Section-isolated state (directory-as-process-boundary)

**What:** Every section gets `.paper/sections/<NN-slug>/` with its own PLAN, DRAFT, VERIFICATION. No section ever reads another section's draft for write-time decisions; the compiler is the only step that opens multiple section directories.

**Why:** This is the load-bearing PRD §4 decision. Re-doing section 3 *cannot* clobber 1/2/4/5 because the writer process never holds those file handles. Bounded verification (~20–40 LLM calls per section vs ~200 for whole paper) is a direct consequence.

**When to use:** Every per-section operation: `plan <N>`, `write <N>`, `verify <N>`. The compiler step (`/pensmith compile`) is the *only* multi-section reader, and it's read-only on `sections/*/DRAFT.md` (it writes only to `.paper/DRAFT.md` and to the *last paragraph* of each section as transition smoothing).

**Trade-offs:**
- (+) Inline conversational corrections become trivial: "redo section 3" maps to a literal directory operation.
- (+) Wave-based parallel writing (Tier 1) works — independent sections share no state.
- (−) Cross-section consistency (e.g., section 2 says X, section 4 says not-X) must be a *post-hoc* compile-time check; can't be detected at write-time. Acceptable per PRD §7.8.
- (−) Section-numbering scheme must support insertion without renumber (PRD §17 recommendation: use folder names like `03-methods/` and `03b-validity-threats/`). **Decision required at planning stage.**

**Prior-art reference:** This is structurally similar to:
- **Pandoc** — each input file is independently parsed before assembly.
- **Sphinx** — each `.rst` source file has its own doctree in `_build/.doctrees/<name>.doctree`; cross-references resolve in a separate phase.
- **Quarto** — each `.qmd` chapter is an independent execution unit; the book is a *project* assembling them. Pensmith's `OUTLINE.md` ↔ Quarto's `_quarto.yml` is a clean analogue.
- **git-annex** — per-file metadata in `.git/annex/objects/<hash>/`; the directory layout is the index.
- **Hugo / Jekyll** — per-page front-matter; site assembly is a build step.

**Example state-isolation guarantee (verified by `tier-contract.test.js`):**

```javascript
// pseudocode — actual test in tests/tier-contract.test.js
test("re-doing section 3 doesn't touch other sections", async () => {
  const before = snapshotAllSectionMtimes(".paper/sections/");
  await runWorkflow("pensmith-plan-section", { N: 3, revise: true });
  await runWorkflow("pensmith-write-section", { N: 3 });
  await runWorkflow("pensmith-verify-section", { N: 3 });
  const after = snapshotAllSectionMtimes(".paper/sections/");
  for (const [path, mtime] of before) {
    if (path.startsWith("03-")) continue;
    expect(after.get(path)).toBe(mtime);
  }
});
```

### Pattern 3: MCP server as state surface, not logic surface

**What:** The MCP server exposes (a) **read-only resources** (`paper://state`, `paper://outline`, `paper://section/{N}`, `paper://library`, `paper://capabilities`) and (b) **idempotent state-mutation tools** (`paper_init_section`, `paper_advance_section`, `paper_record_verification`, `paper_set_status`). Pensmith-specific business logic — DOI verification, claim verification, plagiarism check, humanizer — lives in `bin/lib/` and is also reachable from the CLI. The MCP server merely *brokers* it for the Claude Code session.

**Why:** GSD-plugin's `mcp/server.cjs` does this exact split (it imports `bin/lib/*.cjs` and exposes them as MCP tools). The reason: when business logic lives in MCP, the CLI tier loses it. Keeping logic in `bin/lib/` makes the same logic reachable from `pensmith verify <N>` and from a Claude Code `paper_doi_verify` tool call.

**When to use:** Every MCP tool. The rule: an MCP tool is at most ~30 lines of "validate input → call `bin/lib/<x>.<fn>()` → format response". If a tool is doing work, that work belongs in the library.

**Pensmith-specific MCP surface (recommended):**

```
Resources (read-only):
  paper://state                  — STATE.md frontmatter + section roster
  paper://outline                — OUTLINE.md
  paper://section/{N}            — full section state (PLAN + DRAFT + VERIFICATION)
  paper://section/{N}/plan       — just PLAN.md
  paper://section/{N}/draft      — just DRAFT.md
  paper://section/{N}/verify     — just VERIFICATION.md
  paper://library                — ~/.pensmith/library/index.json
  paper://capabilities           — .paper/CAPABILITIES.json
  paper://config                 — .paper/config.toml as JSON
  paper://research               — RESEARCH.md
  paper://citations              — CITATIONS.bib
  paper://session-log?n=50       — last N entries from SESSION.log

Tools (mutations, all idempotent):
  paper_init_paper(name, class, assignment_prompt)        — creates .paper/, library entry
  paper_init_section(N, slug, plan_stub)                  — creates sections/NN-slug/PLAN.md
  paper_advance_section(N, status)                        — planned → drafted → verified
  paper_record_verification(N, pass1, pass2, pass3, pass4) — writes sections/N/VERIFICATION.md
  paper_set_status(status)                                — intake|research|outline|sectioning|compile|done
  paper_doi_verify(identifier)                            — calls bin/lib/verifier.js, single DOI
  paper_quote_verify(quote, source_id)                    — calls bin/lib/verifier.js
  paper_capability_probe()                                — refreshes CAPABILITIES.json
  paper_record_session(step, inputs, outputs, cost)       — append-only SESSION.log
  paper_lock_acquire() / paper_lock_release()             — for long-running interactive sessions
```

**Trade-offs:**
- (+) CLI tier has zero MCP envy — every capability is reachable both ways.
- (+) MCP server is small enough that a hand-rolled JSON-RPC implementation (gsd-plugin's choice) is realistic; MCP SDK is also fine. Decide at discuss-phase per PRD §17.
- (−) Resource-vs-tool split needs to be carefully maintained: if a "read" resource ever has side effects, the contract is broken.

### Pattern 4: Foundation-first dependency layering

**What:** `bin/lib/` modules form a strict DAG. Foundation modules (paths, atomic-write, lock, http, doi, budget, migrations, pii) have *no dependencies on Domain modules*. Domain modules (state, library, sources, verifier, …) depend only on Foundation. Surface (skills, workflows, agents, MCP server, hooks) depends on Domain.

**Why:** Without this, the build collapses into "I need state to test http, but state depends on http, but I need migrations to test state…". GSD plugin learned this — `bin/lib/core.cjs` is foundational and everything else imports from it.

**Build-order implication (see §7).** Foundation gets its own phase. No feature work begins until atomic-write, paths, http, doi, lock, budget, migrations are unit-tested green.

### Pattern 5: Verifier as deterministic-first, LLM-second

**What:** Pass 1 (DOI integrity, identifier normalisation, author/title fuzzy) and Pass 3 (quote presence in OA full-text) are pure-Node code, not LLM calls. Pass 2 (claim support) and Pass 4 (per-paragraph audit) are LLM-judged but with calibrated prompts and a wave size cap.

**Why:** PRD §14 — "Determinism where it counts." DOI verification cannot be probabilistic; if an LLM "thinks" the DOI exists, the verifier doesn't trust it. `Crossref` says yes or no; `Unpaywall` returns the PDF or doesn't. The LLM is invoked only where judgment is unavoidable.

**When to use:**
- Use deterministic code for: identifier presence, identifier→metadata canonical form, fuzzy string match (Jaro-Winkler / Levenshtein), substring/quote presence in fetched OA text, distinctive-phrase n-gram extraction, plagiarism web-search.
- Use LLM only for: claim-support judgment ({SUPPORTED, PARTIAL, UNSUPPORTED, UNCLEAR}), per-paragraph claim extraction, cross-section consistency contradiction-spotting.

**Trade-offs:**
- (+) `tests/fixtures/known-bad-citations.json` becomes a deterministic acceptance test (10/10 fabricated DOIs flagged, no LLM judgment).
- (+) Cost per section is bounded and predictable (deterministic passes are free; LLM passes capped at wave size 5).
- (−) Fuzzy author/title threshold tuning is a real engineering job — getting it right needs `tests/fixtures/known-good-citations.json` with intentional metadata-quirk cases (Unicode normalisation, "&" vs "and", initial vs full first name).

### Pattern 6: Atomic state writes (write-tmp-then-fsync-then-rename)

**What:** Every state-file write is `fs.writeFileSync(tmp, content); fs.fsyncSync(fd); fs.renameSync(tmp, target)`. The temp file lives alongside the target (same filesystem, rename is atomic) and is named with PID-suffix to allow concurrent writers from different processes.

**Why:** GSD-plugin already does this in `bin/lib/core.cjs:atomicWriteFileSync`. A process killed mid-write must never leave a truncated `STATE.md` that a resume cannot parse.

**When to use:** Every write to:
- `.paper/STATE.md`, `HANDOFF.json`, `CAPABILITIES.json`, `config.toml`
- `.paper/sections/<N>/{PLAN,DRAFT,VERIFICATION}.md`
- `.paper/{RESEARCH,OUTLINE,DRAFT,FINAL,COMPILE-REPORT,VERIFICATION}.md`
- `.paper/CITATIONS.bib`
- `~/.pensmith/library/index.json`

**Exception:** `.paper/SESSION.log` is jsonl append-only — atomic rename per-line is overkill. Use POSIX `O_APPEND | O_DSYNC` write semantics; partial-line on crash is recoverable by the parser (skip incomplete trailing line).

**Implementation:** Lift gsd-plugin's `atomicWriteFileSync` directly into `bin/lib/atomic-write.js`. It already handles cross-device rename failures.

### Pattern 7: Section-granular HANDOFF.json

**What:** The HANDOFF.json schema extends gsd-plugin's 19-field contract with section-granular fields:

```json
{
  "version": "1.0",
  "schema_version": 1,
  "timestamp": "2026-05-06T10:30:00Z",
  "source": "auto-compact",
  "partial": false,

  "paper_name": "transformer-attention-litreview",
  "paper_dir": "/home/user/papers/litreview",
  "status": "sectioning",
  "section": "03-methods",
  "section_phase": "writing",
  "section_progress": { "current": 3, "total": 7, "wave": "B" },

  "current_step": "pensmith-write-section",
  "current_substep": "drafting paragraph 4 of 8",
  "next_action": "Resume write-section 3 from paragraph 4",

  "completed_sections": [
    { "n": "01-introduction", "status": "verified" },
    { "n": "02-background",  "status": "verified" }
  ],
  "remaining_sections": [
    { "n": "04-results", "status": "planned" },
    { "n": "05-discussion", "status": "outlined" }
  ],

  "blockers": [],
  "human_actions_pending": [
    { "kind": "approve_outline", "blocked_at": "2026-05-06T10:15:00Z" }
  ],
  "decisions": [],
  "uncommitted_files": [
    { "status": "modified", "path": ".paper/sections/03-methods/DRAFT.md" }
  ],
  "running_cost_usd": 1.42,
  "context_notes": "User asked for 1500 words instead of 2500; section targets adjusted."
}
```

**Why:** Resume must restart at the right *section* and the right *step within that section*. Restart granularity = section + (plan|write|verify) + paragraph index when mid-write.

**Trade-offs:**
- (+) `/compact` recovery resumes at the substep, not the start of the section.
- (+) Backwards-compatible with gsd-plugin's HANDOFF schema (we keep the 17 required upstream fields, extend additively).
- (−) PreCompact hook has 5s budget; gathering section state must be fast. Cache: section-list comes from `glob(.paper/sections/*/DRAFT.md)`, status from frontmatter — both <100ms.

### Pattern 8: Concurrent-run lock (PID + start-time, stale auto-clear)

**What:** `.paper/.pensmith.lock` contains `{pid, started_at, hostname, longest_step_timeout}`. New session: read lock; if PID alive AND `now - started_at < longest_step_timeout`, refuse with `"another pensmith session is running (PID 12345, started 4 minutes ago)"`. Else clear and acquire.

**Why:** Two `/pensmith` sessions writing to the same `.paper/sections/03-methods/DRAFT.md` is a corruption disaster. The lock file is a sentinel; PID liveness check (`kill(pid, 0)`) handles the common case (user closed terminal); timeout handles the edge case (process killed without cleanup, PID was reused by an unrelated process).

**Where it goes:** `bin/lib/lock.js`. Acquire on workflow entry, release on Stop hook (T1) or `process.on('exit', …)` (T2).

### Pattern 9: Cost cap as cross-cutting concern

**What:** `bin/lib/budget.js` wraps every LLM call (T2 `runtime.js`) and every paid API call. It exposes:
- `budget.estimate(step) → usd` — pre-step projection.
- `budget.charge(usd, step)` — running meter; throws if would exceed `cost_cap_usd`.
- `budget.summary() → { total, byStep, remaining }` — for `/pensmith status`.

**Why:** PRD §14 mandates a hard abort. The only way that's airtight is for *every* paid call to flow through one chokepoint. Like `bin/lib/http.js` for source APIs.

**When invoked:** Around every `runtime.complete()` call, around every `honesty.score()`, around every paid `sources.*` call. The pre-call estimate prevents starting a 10K-token completion with $0.30 of budget left.

### 4.1 End-to-end happy path (one paper, intake → done)

```
[user: /pensmith @assignment.pdf]
    │
    ├─► intake (pensmith-intake skill / pensmith-new workflow)
    │     ▶ AskUserQuestion (T1) / stdin loop (T2): discipline, mode, goal, class, counterargument, style-match, PII

    │     ▶ disclaimer printed
    │

    │     ▶ disambiguation pass (tiny subagent if ambiguous terms detected)
    │     ▶ generate 5–10 queries from PROJECT.md
    │     ▶ <capability_check> Task available → parallel pensmith-source-researcher
    │                          else            → sequential loop
    │     ▶ if BYO PDFs → bin/lib/pdf-ingest.js: parse + extract metadata + Crossref hydrate
    │     ▶ if Zotero MCP detected + authed → pull collection
    │     ▶ pensmith-source-evaluator scores, dedupes, tiers
    │     ▶ approval gate (AskUserQuestion / stdin)
    │     ▶ writes .paper/RESEARCH.md, .paper/CITATIONS.bib (with last_verified timestamps)
    │     ▶ status → research
    │
    ├─► outline (pensmith-outline workflow)
    │     ▶ pensmith-outliner produces section structure with thesis, mappings, dependencies
    │     ▶ if counterargument required → refuse outline without counter section
    │     ▶ approval gate
    │     ▶ writes .paper/OUTLINE.md
    │     ▶ creates .paper/sections/<NN-slug>/PLAN.md per section (stub from outline entry)
    │     ▶ status → sectioning
    │
    ├─► for each section in topological order (Tier 1: parallel waves; Tier 2: sequential):
    │     │
    │     ├─► plan-section (pensmith-plan-section workflow)
    │     │     ▶ pensmith-section-planner reads stub PLAN.md
    │     │     ▶ maps claims → sources, identifies counterexamples, paragraph structure
    │     │     ▶ writes .paper/sections/<N>/PLAN.md
    │     │
    │     ├─► write-section (pensmith-write-section workflow)
    │     │     ▶ pensmith-section-writer prompt receives ONLY this section's mapped sources
    │     │     ▶ if style-match enabled → consume .paper/STYLE.json
    │     │     ▶ writes .paper/sections/<N>/DRAFT.md
    │     │     ▶ auto-chains to verify-section (unless --no-verify)
    │     │
    │     └─► verify-section (pensmith-verify-section workflow) — bounded
    │           ▶ Pass 1 (deterministic): bin/lib/verifier.js
    │           │   – extract DOIs/arXiv/PMIDs from DRAFT.md
    │           │   – bin/lib/doi.js normalise
    │           │   – bin/lib/sources.js fetch via Crossref/arXiv/PubMed
    │           │   – 404 → FABRICATED; metadata mismatch → MIS-CITED
    │           ▶ Pass 2 (LLM): pensmith-claim-verifier per citation, waves of 5
    │           │   – verdict ∈ {SUPPORTED, PARTIAL, UNSUPPORTED, UNCLEAR}
    │           ▶ Pass 3 (deterministic): for each direct quote, fetch OA via Unpaywall, substring + fuzzy match
    │           ▶ Pass 4 (LLM): pensmith-paragraph-auditor — orphan-claim detection
    │           ▶ writes .paper/sections/<N>/VERIFICATION.md
    │           ▶ section marked verified iff Pass 1 clean + Pass 3 clean (FABRICATED=0, MIS-CITED=0, NOT_FOUND=0)
    │
    ├─► compile (pensmith-compile workflow)
    │     ▶ refuses if any section has FABRICATED / MIS-CITED / NOT_FOUND
    │     ▶ pensmith-compiler reads all sections/<N>/DRAFT.md
    │     ▶ concatenates in OUTLINE.md order
    │     ▶ cross-section smoothing pass (last ¶ of section i + first ¶ of section i+1 only)
    │     ▶ cross-section claim consistency check (LLM scans for X / not-X across sections)
    │     ▶ citation-density check vs discipline preset target
    │     ▶ writes .paper/DRAFT.md, .paper/COMPILE-REPORT.md
    │     ▶ status → compile
    │
    └─► done (pensmith-done workflow)
          ▶ refuses if any section unclean
          ▶ whole-paper Pass 4 (per-paragraph audit) on .paper/DRAFT.md
          ▶ bin/lib/plagiarism.js: distinctive-phrase n-grams → DuckDuckGo HTML
          ▶ <capability_check> humanizer skill present → invoke; else skip cleanly
          ▶ bin/lib/honesty.js: GPTZero/Originality/Sapling score (before AND after humanize)
          ▶ approval gate (skip with --yolo)
          ▶ <capability_check> Pandoc → .docx/.pdf/.tex; else .md
          ▶ writes .paper/FINAL.md, .paper/VERIFICATION.md, exports
          ▶ NO metadata stamp in exports (deliberate)
          ▶ status → done
```

### 4.2 State-mutation flow (every step)

```
workflow step
    │
    ├─► acquire lock (bin/lib/lock.js) — first step only
    ├─► read STATE.md frontmatter (bin/lib/state.js)
    ├─► budget pre-check (bin/lib/budget.js.estimate)
    ├─► (work happens — LLM calls / API calls / file writes)
    ├─► budget.charge(actual)
    ├─► atomic-write any state files (bin/lib/atomic-write.js)
    ├─► append SESSION.log entry (bin/lib/session-log.js)
    ├─► if running > 60s since last checkpoint → write HANDOFF.json (bin/lib/checkpoint.js)
    └─► (release lock at workflow end)
```

### 4.3 Resume flow

```
SessionStart hook fires
    │
    ▼
hooks.json runs `node bin/pensmith-tools.js hook session-start`
    │
    ▼
bin/pensmith-tools.js:
    1. find .paper/ (cwd or library lookup)
    2. read HANDOFF.json
    3. if absent → pensmith --status
       else → emit resume-prompt to Claude:
              "Last checkpoint: section 03-methods, write-section step,
               paragraph 4 of 8. Resume? [Y/n]"
    │
    ▼
user confirms (or skipped with --yolo) → pensmith-resume skill →
    pensmith.md umbrella workflow re-enters at the recorded substep
```

### 4.4 Wave-based parallel writing (Tier 1 only)

```
OUTLINE.md declares per-section depends_on: [N, M, …]
    │
    ▼
bin/lib/state.js.computeWaves():
    – topological sort
    – wave 0: sections with no deps
    – wave k+1: sections whose deps are all in waves ≤ k
    │
    ▼
<capability_check> Task available?
    yes → spawn one pensmith-section-planner subagent per section in current wave
          await all
          spawn one pensmith-section-writer per section in current wave
          await all
          spawn one pensmith-verify-section per section (bounded subagent set: doi, claim, quote, paragraph)
          await all
          advance to next wave
    no  → sequential loop in topological order; wave structure ignored
```

## 6. Scaling considerations

| Scale | Architecture adjustments |
|-------|--------------------------|
| **1 paper, 5 sections** (the common case) | No adjustments. Vanilla workflow. |
| **10 papers in library** | Library index.json starts to grow; still under 10KB, fine as JSON. `/pensmith list` is instant. |
| **100 papers in library** | JSON index still fine (~100KB). If `/pensmith list` becomes slow, consider SQLite index (PRD §17 open question). Decision deferred. |
| **1 paper, 30 sections** (a thesis chapter set) | Wave scheduling matters. `bin/lib/state.js.computeWaves()` is O(V+E) topological sort — trivial. Watch for verifier cost: 30 sections × 25 LLM calls = 750 calls. Cost cap will trip. Mitigation: per-section budget pre-check refuses wave start if remaining budget < estimated wave cost. |
| **Bursty source-API load** | `bin/lib/http.js` already handles: per-source TTL cache, exponential backoff with jitter, polite User-Agent. OpenAlex / Crossref are generous; arXiv has stricter rate limits — set per-source `min_request_interval_ms`. |
| **Long sections (5K+ words)** | LLM context window is the bottleneck, not pensmith. Section writer's prompt envelope = (section PLAN + mapped sources' abstracts + style profile + voice hints). Mapped sources' full text (when OA) is loaded *only* during verify Pass 3. |

### Scaling priorities

1. **First bottleneck: cost cap trips on multi-section verify.** Mitigation: precise per-pass cost estimate; allow `--budget-section <N>` override; per-section dry-run that surfaces "section 5 estimate $0.80, current remaining $0.40 — abort wave?" before spawning.
2. **Second bottleneck: source-API rate limits during research.** Mitigation: HTTP cache TTL (24h DOI, 1h search), exponential backoff with jitter (already in PRD §10), `PENSMITH_CONTACT_EMAIL` for polite UA.
3. **Third bottleneck: PDF parsing fidelity for BYO PDFs.** PRD §17 leaves the choice (`pdf-parse` pure JS vs shell to `pymupdf`). Recommendation: ship `pdf-parse` as default; if `pymupdf` detected on PATH, prefer it. Mirrors ecosystem-composition pattern.

## 8. Anti-patterns

### Anti-pattern 1: "Logic in the MCP server"

**What people do:** Implement DOI verification or claim-judgment inside `mcp/server.js` as MCP tools.
**Why it's wrong:** Tier 2 has no MCP. The CLI loses the capability. `tier-contract.test.js` fails — not because the test is broken, but because the architecture is.
**Do this instead:** All logic lives in `bin/lib/*`. MCP tools are 30-line shims that validate input and call the library.

### Anti-pattern 2: "Skill body has the steps, workflow body is empty"

**What people do:** Write the actual procedure in `skills/<name>/SKILL.md` and leave `workflows/<name>.md` as a placeholder.
**Why it's wrong:** Tier 2 doesn't read SKILL.md (it has no skill resolution). Both tiers must read the workflow body. Logic in SKILL.md is unreachable from CLI.
**Do this instead:** SKILL.md is frontmatter + `@${CLAUDE_PLUGIN_ROOT}/workflows/<name>.md` include only. All steps live in the workflow body.

### Anti-pattern 3: "Cross-section read at write-time"

**What people do:** During `write-section 3`, prompt-stuff the writer with sections 1 and 2's drafts for "context."
**Why it's wrong:** Breaks the section-as-phase isolation. Re-doing section 2 *now* invalidates the cache assumption section 3's writer made. Verifier scope explodes.
**Do this instead:** Use the OUTLINE.md's per-section thesis + dependency declarations as the only cross-section context at write-time. Cross-section reading happens *only* in the compile step.

### Anti-pattern 4: "LLM-judged DOI verification"

**What people do:** Ask an LLM "does this DOI look real?" or "do these authors match this paper?".
**Why it's wrong:** Probabilistic answer to a deterministic question. Hallucinated yes is the worst possible failure mode.
**Do this instead:** Pass 1 is `Crossref` HTTP fetch + Jaro-Winkler on the returned canonical metadata. LLM is invoked *only* for claim-support judgment (Pass 2), where judgment is unavoidable.

### Anti-pattern 5: "Forgetting `<capability_check>` and assuming Task is available"

**What people do:** Write `workflows/pensmith-research.md` with "spawn N source-researcher subagents" and no fallback.
**Why it's wrong:** Tier 2 has no Task. Workflow body fails or silently degrades to a broken state.
**Do this instead:** Every step that uses Task / MCP / AskUserQuestion / Pandoc / Zotero MCP / humanizer declares the capability at the top of the workflow body and provides an inline fallback. `tests/tier-contract.test.js` will catch missing fallbacks.

### Anti-pattern 6: "Renumbering sections on insert"

**What people do:** User says "add a section between 3 and 4" → tool renumbers 4→5, 5→6, … and renames `04-discussion/` to `05-discussion/`.
**Why it's wrong:** Breaks every checkpoint, every HANDOFF.json, every SESSION.log entry that references "section 04". Resume becomes ambiguous.
**Do this instead:** Stable numbering. Insert as `03b-validity-threats/` (PRD §17 recommendation). The folder name lexicographically sorts between `03-methods/` and `04-discussion/`. OUTLINE.md tracks logical order; folder name is just an ID.

### Anti-pattern 7: "Sync writes mid-section"

**What people do:** During section drafting, write each paragraph to disk as it's generated.
**Why it's wrong:** Mid-write crashes leave a half-section. Atomic-write is per-file; partial-section is meaningless.
**Do this instead:** Buffer the section draft in memory; atomic-write the whole DRAFT.md on completion. Use HANDOFF.json (mtime-throttled) to record paragraph-index progress for resume — *not* the partial draft itself.

### Anti-pattern 8: "Verifier as soft warning"

**What people do:** Surface FABRICATED / NOT_FOUND as warnings the user can dismiss; allow compile/export to proceed.
**Why it's wrong:** PRD core value: "no FABRICATED, MIS-CITED, or quote-NOT_FOUND ever escapes a section." Soft warnings violate the contract.
**Do this instead:** Hard refuse at compile and export. User must `plan <N> --revise` and re-write. The only escape hatch is `--yolo`, which is documented as deliberately gated.

### Internal boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| Skill ↔ workflow body | `@${CLAUDE_PLUGIN_ROOT}/workflows/<name>.md` include | One-way; skill is frontmatter only |
| Workflow body ↔ agent | Agent prompt loaded from `agents/<name>.md`; T1 spawns via Task; T2 sequential | Same prompt envelope both tiers |

| T2 CLI ↔ `bin/lib/*` | Direct `require()` | Process exits cleanly; release lock via `process.on('exit')` |

## 10. Build-order implications (for downstream roadmap consumer)

The roadmap should structure around these phase boundaries. Each is a natural commit-and-validate point:

1. **Foundation NFRs** is its own phase, with sub-phases or task batches per Foundation lib if needed. **No feature phase begins until Foundation is green.** This is non-negotiable per PRD §19.
2. **Tier-1 shell + Tier-2 shell** can run in parallel after Foundation, but `tests/tier-contract.test.js` must exist and pass on `doctor` as the first contract case before any feature workflow is added.
3. **Vertical slice through one section** is the proof phase. It validates the entire architecture (skills, workflows, agents, MCP, hooks, atomic state, capability checks, verifier deterministic passes) against a single fixture. If this phase exposes drift, foundation gets revised before breadth begins.
4. **Wave 4–10 phases** are widening, not deepening. Each adds breadth (more sections, more verifier passes, more disciplines, more citation styles, more ecosystem hooks) on top of the proven slice.
5. **`tier-contract.test.js` is the architectural gate.** It runs every workflow body in both tiers against the same fixtures. New workflows don't merge until they're in the contract test.

Tier 1 / Tier 2 split visible at component level: see §5 table. The split is *strictly* at the surface layer — every domain library and every foundation library is shared. Tier 1 adds: plugin manifest, MCP server, hooks, skills/SKILL.md frontmatter. Everything else is one source of truth.
