<objective>
Ship the global paper registry (LIB-01) and the `list` (LIB-02) + `open` (LIB-03) verbs, with the LIB-05 status lifecycle DERIVED at display time (LIB-05). The registry is a NEW file at pensmithDataDir()/library/index.json with its OWN zod schema — strictly separate from the per-paper .paper/LIBRARY.json (D-59 source/citation store). It RETAINS each paper's absolute folderPath so `open` can switch to it and `list` can load each paper's STATE.json to compute its live status. Promote list/open from Phase-2 stubs via REAL_VERB_LOADERS (no 17th verb).

LIB-05 status-writer decision (resolves 08-RESEARCH Open-Q4 — DERIVE not UPSERT): rather than having all six lifecycle verbs (research/outline/plan/write/compile/done) UPSERT a `status` + `sectioningProgress` into the registry on every transition (the verbs no plan touches, and which would go stale the instant a paper advances out-of-band), `list` DERIVES each paper's lifecycle value at display time from that paper's AUTHORITATIVE STATE.json + section PLAN.md frontmatter — the same on-disk state machine resolveNextAction already walks. Single source of truth (STATE.json), zero churn across six verbs, and the `sectioning X/Y` branch is a live computed value, not dead code. The stored `entry.status` is written once at intake registration (08-05) and is consulted ONLY for the terminal `archived` flag (which has no on-disk STATE.json marker and is therefore the one state that cannot be derived).

Purpose: Enable multi-paper management — see every paper grouped by class WITH its real lifecycle status, switch the active paper without `cd`. This is the SC-1 backbone.
Output: 2 new schema/lib files (incl. the deriveLibraryStatus helper), 3 new paths.ts exports, 2 promoted CLI verbs, 2 REAL_VERB_LOADERS entries.
